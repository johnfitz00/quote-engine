# JPA / Hibernate vs Eloquent

JPA (Java Persistence API) is the standard database abstraction in Java/Kotlin. Hibernate is the most common implementation. Together they're roughly equivalent to Laravel's Eloquent — but with a very different philosophy.

---

## The fundamental difference: Data Mapper vs Active Record

| | Eloquent (Laravel) | JPA / Hibernate |
|-|-------------------|-----------------|
| Pattern | Active Record | Data Mapper |
| Model queries itself | Yes — `Quote::find($id)` | No — repository does it |
| Model knows about the DB | Yes | Ideally no |
| Separate repository | Optional | Standard practice |

**Active Record:** the model class contains both the data *and* the database operations. `Quote::create()`, `$quote->save()`, `Quote::where()->get()` all live on the model.

**Data Mapper:** the model contains only data. A separate `QuoteRepository` class handles reading/writing. The domain object (`Quote`) doesn't know it's being persisted.

---

## `@Entity` vs an Eloquent model

```kotlin
// Quote.kt
@Entity
class Quote {
    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    var id: UUID? = null

    var policyHolderName: String? = null

    @Embedded
    var vehicle: Vehicle? = null

    @Embedded
    var driver: Driver? = null

    var status: QuoteStatus = QuoteStatus.DRAFT

    @CreationTimestamp
    var createdAt: Instant? = null

    @UpdateTimestamp
    var updatedAt: Instant? = null
}
```

```php
// Laravel
class Quote extends Model {
    protected $fillable = ['policy_holder_name', 'status'];
    protected $casts = [
        'status' => QuoteStatus::class,
        'created_at' => 'datetime',
    ];
}
```

`@Entity` marks this class as a JPA-managed object that maps to a database table. The table name defaults to the class name (`quote`). There's no `$table` property — convention rules.

Key differences:
- **No `$fillable`** — mass assignment protection doesn't exist in JPA. You control what gets set in your mapper/service code
- **No `$casts`** — types are handled by the type system itself (`QuoteStatus` enum, `Instant` for timestamps)
- **`@CreationTimestamp` / `@UpdateTimestamp`** — Hibernate sets these automatically, equivalent to Eloquent's `$timestamps = true`

## Why fields start as `null`

```kotlin
var id: UUID? = null
var policyHolderName: String? = null
var createdAt: Instant? = null
```

JPA (Hibernate) requires a **no-argument constructor** so it can construct an empty object and then populate it via reflection when loading from the database. So when Hibernate reads a row, it does:

```kotlin
val quote = Quote()          // empty object, everything null
quote.id = resultSet.getUUID("id")
quote.policyHolderName = resultSet.getString("policy_holder_name")
// etc.
```

That's why the fields start nullable even though in practice they'll always have values after being loaded. The `!!` assertions in `QuoteMapper.toResponse()` say "I know this will be populated by now."

## `@Embedded` vs a JSON column or separate table

```kotlin
// Quote.kt
@Embedded
var vehicle: Vehicle? = null

// Vehicle.kt
@Embeddable
data class Vehicle(
    @Column(name = "vehicle_year")
    val year: Int,
    val make: String,
    val model: String,
    val annualKm: Int
)
```

`@Embedded` means Vehicle's fields are flattened into the **same table** as Quote — no separate `vehicles` table, no foreign key. The `quote` table will have columns: `vehicle_year`, `make`, `model`, `annual_km`.

```php
// Laravel equivalent options:
// Option 1: JSON column
protected $casts = ['vehicle' => 'array'];

// Option 2: separate table with belongsTo
class Quote extends Model {
    public function vehicle(): HasOne { return $this->hasOne(Vehicle::class); }
}
```

`@Embeddable` is the marker on the class that *can* be embedded. `@Embedded` is the marker on the field that *uses* an embeddable. The `@Column(name = "vehicle_year")` override is needed because `year` would clash with a SQL reserved word.

## Repository pattern: the interface Spring generates

```kotlin
interface QuoteRepository : JpaRepository<Quote, UUID> {
    fun findByStatus(status: QuoteStatus): List<Quote>
}
```

You write only the interface. Spring Data generates the full implementation at startup:

| Method | Generated by |
|--------|-------------|
| `findById(id)` | `JpaRepository` — always provided |
| `findAll()` | `JpaRepository` — always provided |
| `save(entity)` | `JpaRepository` — always provided |
| `saveAndFlush(entity)` | `JpaRepository` — always provided |
| `deleteAll()` | `JpaRepository` — always provided |
| `findByStatus(status)` | Spring Data — reads the method name |

**Method name conventions** (Spring Data derives SQL from them):

```kotlin
fun findByStatus(status: QuoteStatus)              // WHERE status = ?
fun findByStatusAndPolicyHolderName(...)           // WHERE status = ? AND policy_holder_name = ?
fun findByAgeGreaterThan(age: Int)                 // WHERE age > ?
fun findAllByOrderByCreatedAtDesc()                // ORDER BY created_at DESC
```

The naming convention is rigid but powerful — no SQL for simple queries.

## `findById` returns `Optional`, not the entity directly

```kotlin
val quote = quoteRepository.findById(id)
    .orElseThrow { QuoteNotFoundException(id) }
```

`findById()` returns `Optional<Quote>` — a wrapper that either contains the entity or is empty. This is Java's way of forcing you to handle the "not found" case at compile time.

| Java/Kotlin | PHP equivalent |
|-------------|----------------|
| `Optional.orElseThrow { }` | `findOrFail()` |
| `Optional.orElse(null)` | `find()` (returns null) |
| `Optional.isPresent()` | `$result !== null` |
| `Optional.get()` | `$result` (unsafe) |

Laravel's `findOrFail()` does the same in one step. JPA splits it into `findById()` returning Optional, then you decide what to do with the empty case.

---

## `save` vs `saveAndFlush`

```kotlin
quoteRepository.saveAndFlush(quote)  // used in this project
quoteRepository.save(quote)
```

Within a transaction, JPA batches SQL writes and flushes them to the database at the end. `save()` queues the write. `saveAndFlush()` forces the write immediately.

This project uses `saveAndFlush` because `@CreationTimestamp` and `@UpdateTimestamp` are set by the database, and we need those values back (in `toResponse()`) before the transaction ends. Without flush, `createdAt` would still be `null`.

In Eloquent, `$model->save()` always writes immediately — there's no lazy flush concept.

## Enum mapping

```kotlin
var status: QuoteStatus = QuoteStatus.DRAFT
```

By default, Hibernate stores the enum as a string (the name: `"DRAFT"`, `"RATED"`, etc.). No annotation needed for this default. If you wanted to store as an integer instead, you'd add `@Enumerated(EnumType.ORDINAL)`.

```php
protected $casts = ['status' => QuoteStatus::class]; // Laravel 9+
```

Both store by name by default.

---

## Timestamps: `@CreationTimestamp` and `@UpdateTimestamp`

```kotlin
@CreationTimestamp
var createdAt: Instant? = null

@UpdateTimestamp
var updatedAt: Instant? = null
```

Hibernate sets `createdAt` on first persist, `updatedAt` on every update. This is exactly `$timestamps = true` in Eloquent (the `created_at`/`updated_at` columns).

`Instant` is a timezone-less point in time (epoch millis). It's the Java equivalent of PHP's `\DateTimeImmutable` or Carbon. Eloquent uses Carbon; JPA uses `Instant`, `LocalDateTime`, or `ZonedDateTime`.

## The `allOpen` plugin in `build.gradle.kts`

```kotlin
// build.gradle.kts
allOpen {
    annotation("jakarta.persistence.Entity")
    annotation("jakarta.persistence.MappedSuperclass")
    annotation("jakarta.persistence.Embeddable")
}
```

Kotlin classes are `final` by default. Hibernate needs to subclass your entity to create "proxy" objects for lazy loading. Without `allOpen`, Hibernate would crash because it can't extend a final class.

This plugin automatically marks any class annotated with `@Entity`, `@Embeddable`, etc. as `open`. You never see `open` on entity classes in the source code — the plugin adds it at compile time.

The `plugin.jpa` plugin (also in `build.gradle.kts`) adds a no-argument constructor to every `@Entity` class — which JPA requires, as explained earlier.

## Lazy loading — why this project avoids it

JPA's default behaviour for `@OneToMany` relationships is **lazy loading**: the related data isn't fetched from the database until you access it. This can cause the "N+1 problem" and, worse, can cause errors if you try to access lazy data after the transaction has closed.

This project avoids the problem entirely by using `@Embedded` (not a separate table/relationship) and by calling `toResponse()` inside the service (inside the transaction). By the time the response leaves the service, everything is already plain data — no JPA proxies, no lazy loading risk.

The criterion "QuoteResponse contains no lazy-loaded JPA proxies" is satisfied because the mapper converts everything to plain `data class` objects before returning from the service.
